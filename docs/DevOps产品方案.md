## 1. 产品概述

### 1.1 产品定位

**ServerPilot** 是一款**开源的** AI 驱动服务器运维平台。用户通过**自然语言对话**管理服务器，AI 自动将运维意图转化为可执行的操作计划，由部署在目标服务器上的 Agent 安全执行。

**一句话描述**: "跟 AI 聊天就能管服务器 — 开源、安全、自主可控。"

**市场定位**: 宝塔面板的 AI 时代替代品。同样一行命令安装，但用 AI 对话取代表单点击，用开源透明取代闭源黑盒。

### 1.2 核心理念

```
传统运维: 用户 → 写脚本/敲命令 → 服务器
ServerPilot: 用户 → 对话 AI → AI 生成计划 → Agent 执行 → 结果反馈
```

- **服务端(Server)** = 大脑 — 理解意图、制定计划、调度执行
- **客户端(Agent)** = 手脚和眼睛 — 探测环境、执行命令、汇报状态
- **AI Provider** = 可插拔的智能引擎 — 用户自选模型 (Claude / GPT / DeepSeek / Ollama 本地模型)

### 1.3 解决的核心问题

| 痛点 | ServerPilot 的解决方案 |
|------|----------------------|
| 运维门槛高，需要记大量命令 | 自然语言对话，AI 自动生成命令 |
| 多台服务器管理分散 | 统一 Web 面板，集中管控 |
| 定时任务配置复杂 (crontab) | 对话创建: "每天凌晨3点备份数据库" |
| 故障排查耗时 | AI 实时诊断，自动给出修复方案 |
| 操作缺乏记录和审计 | 全量操作日志，可追溯、可回放 |
| 环境差异导致操作失败 | Agent 自动探测环境，AI 适配生成命令 |
| 不信任第三方工具操作生产服务器 | 开源代码、本地部署、数据自主可控 |
| 每次操作都要重复描述服务器情况 | 服务器独立档案，AI 记住每台机器的一切 |

**为什么选择 Open Core:**

| 对比维度 | 纯闭源 SaaS | Open Core |
|---------|------------|-----------|
| 用户信任 | 低 (代码不可见) | 高 (代码透明，可审计) |
| 获客成本 | 高 (需要广告投放) | 低 (社区自然增长) |
| 冷启动速度 | 慢 (付费门槛) | 快 (免费试用零门槛) |
| 社区效应 | 无 | GitHub Stars + 贡献者 + 口碑传播 |
| 适合运维工具 | 差 (用户不敢在生产服务器装不透明的工具) | 好 (开源 = 信任) |

**增长飞轮:**

```
开源免费 → 开发者试用 (零成本)
  → 好用 → GitHub Star + 口碑分享
    → 更多人试用 → 社区贡献 → 产品更好
      → 部分用户需要团队/高级功能 → 付费转化云版
```

## 2. 目标用户

### 2.1 核心用户画像

#### 画像 A: 独立开发者 / 小团队 CTO

- 有 1-5 台服务器(VPS/云主机)
- 技术能力参差不齐，运维不是主业
- 期望: 花最少时间保证服务器稳定运行
- **开源版**: 自部署，自带 API Key (零成本)
- **云版付费意愿**: $10-30/月

#### 画像 B: 初创公司运维工程师

- 管理 5-20 台服务器
- 有运维基础但希望提效
- 期望: 自动化重复工作、减少人为失误
- **开源版**: 自部署，可能贡献插件
- **云版付费意愿**: $30-100/月

#### 画像 C: 中小企业 IT 部门

- 管理 20-100 台服务器
- 需要审计、合规、权限管理
- 期望: 统一管控、降低运维人力成本
- **直接使用云版企业版**
- **付费意愿**: $100-500/月

### 2.2 非目标用户 (当前阶段)

- 大型企业 (1000+ 服务器，已有成熟运维体系)
- 纯 Kubernetes/Serverless 架构 (无传统服务器)

## 3. 核心价值主张

### 3.1 对话即运维

```
用户: "帮我在 prod-01 上安装 Nginx，配置反向代理到 3000 端口"

AI: 我来帮你完成这个任务，需要以下步骤:
  1. 安装 Nginx (apt install nginx)
  2. 创建配置文件 (/etc/nginx/sites-available/app)
  3. 配置反向代理 (proxy_pass http://localhost:3000)
  4. 启用站点并重启 Nginx

  预计耗时 2 分钟，是否执行？

用户: 执行

AI: ✅ 已完成！Nginx 已安装并配置好反向代理。
    访问 http://prod-01-ip 即可验证。
```

### 3.2 服务器档案 — AI 记住每台服务器的一切

每台服务器都拥有一份独立的**智能档案 (Server Profile)**，AI 在每次对话时自动加载，无需用户重复描述服务器情况。

```
没有档案时的对话:
  用户: "帮我部署一个新项目"
  AI: "请问你的服务器是什么系统? 装了什么软件? 用什么方式部署?..."
  (每次都要重新了解，体验差)

有档案后的对话:
  用户: "帮我部署一个新项目"
  AI: "好的，我看到 prod-01 上已经有 Docker 和 Nginx。
       我建议用 Docker Compose 部署，Nginx 反向代理到新容器。
       上次你的 myapp 也是这样部署的，保持一致。
       备份目录我用 /backup，和之前一样。
       需要执行吗?"
  (AI 记住一切，像一个了解你服务器的专属运维专家)
```

详细设计见 [4.8 服务器档案系统](#48-服务器档案系统-server-profile)。

### 3.3 自成长知识库 — 用得越多，AI 越准

AI 对 Nginx、MySQL 这些主流软件很熟悉，但遇到小众软件或新项目就可能出错。ServerPilot 的知识库会**自动积累经验**:

```
知识来源:
  ① 内置: 产品自带常见软件的安装知识 (Nginx, MySQL, Docker...)
  ② 自动学习: 每次成功操作自动沉淀为知识，下次复用
  ③ 自动抓取: 遇到未知软件时，自动去 GitHub/官网抓取安装文档
  ④ 社区共享: 用户可分享脱敏后的操作经验 (opt-in)

效果:
  第一次装 Caddy → AI 去 GitHub 抓取文档 → 成功 → 自动沉淀
  第二次装 Caddy → 直接复用上次验证过的命令 → 更快更准

  10 个用户在不同平台装过同一软件 → 聚合出最全面的知识
  → 越多人用 → 知识库越丰富 → AI 越准 → 更多人用 (飞轮)
```

详细设计见 [13.2 知识库累积体系](#132-知识库累积体系)。

## 4. 产品架构总览

### 4.1 部署模式

ServerPilot 采用统一代码库，Server 和 Agent 都安装在用户自己的服务器上 (社区版)，也可通过云服务托管 Server (商业版)。

#### 自部署模式 A: 单机一体 (推荐新手，管理 1 台服务器)

**Server (含 Dashboard) + Agent 全部装在同一台服务器上，一条命令搞定。**

```
┌──────────────────────────────────────┐
│  用户的服务器                          │
│                                      │
│  ┌────────────────────────────────┐  │
│  │  Docker Compose                │  │
│  │                                │  │
│  │  ┌────────────────────────┐   │  │
│  │  │ Server (:3000)         │   │  │
│  │  │  · Dashboard (React)   │   │  │
│  │  │  · REST API            │   │  │
│  │  │  · AI 引擎 + WSS      │   │  │
│  │  │  · SQLite DB           │   │  │
│  │  └───────────┬────────────┘   │  │
│  │              │ (本机通信)       │  │
│  │  ┌───────────┴────────────┐   │  │
│  │  │ Agent (守护进程)        │   │  │
│  │  │  · 环境探测 + 命令执行  │   │  │
│  │  └────────────────────────┘   │  │
│  └────────────────────────────────┘  │
│                                      │
│  AI Provider: 用户自选 (API Key)      │
└──────────────────────────────────────┘

安装: curl -fsSL https://get.serverpilot.dev | bash
使用: 浏览器打开 http://服务器IP:3000，直接开始对话

→ 适合: 只有 1 台服务器的个人开发者
→ 一条命令部署，立即可用，数据全在本机
```

#### 自部署模式 B: 一主多从 (管理多台服务器)

**选一台服务器装 Server (含 Dashboard)，其余服务器只装 Agent。**

```
┌──────────────────────────────────┐
│  服务器 A (管理节点)               │
│  Server (含 Dashboard) + Agent   │
│  :3000 (Dashboard + API + WSS)   │
└──────────────┬───────────────────┘
               │ WSS (内网或公网)
       ┌───────┼───────┐
  ┌────┴───┐  ┌┴──────┐  ┌────┴───┐
  │ 服务器B │  │服务器C │  │服务器D  │
  │ Agent  │  │Agent  │  │Agent   │
  └────────┘  └───────┘  └────────┘

管理节点安装: curl -fsSL https://get.serverpilot.dev | bash
其他节点安装: curl -fsSL https://get.serverpilot.dev/agent | bash
Agent 连接:   serverpilot-agent --server wss://服务器A:3001

→ 适合: 有 2-20 台服务器的团队
→ 所有数据集中在管理节点上
→ Agent 之间互不影响
```

#### 自部署模式 C: 本机管远程 (从笔记本管理云服务器)

```
┌──────────────────┐
│  用户的 Mac/PC    │
│  Server (含 Dashboard) │
│  :3000            │
└────────┬─────────┘
         │ WSS (公网)
    ┌────┼────┐
  ┌─┴──┐ ┌─┴──┐ ┌──┴─┐
  │云主机│ │云主机│ │云主机│
  │  A  │ │  B  │ │  C  │
  │Agent│ │Agent│ │Agent│
  └────┘ └────┘ └────┘

→ 适合: 从本地管理远程云服务器
→ 需要 Agent 能连到本机 (公网 IP / 内网穿透 / VPN)
→ Server 关机时 Agent 离线，重启后自动重连
```

#### 云服务模式 (商业版)

```
┌─────────────────┐     ┌──────────────────────────┐
│  Web Dashboard    │◄──►│    ServerPilot Cloud      │
│  app.serverpilot  │     │  · 用户管理 / 多团队      │
│  .dev             │     │  · 托管 AI (用户免配置)    │
│                   │     │  · 服务器档案云端同步      │
│                   │     │  · 高级功能 (审计/合规)    │
└─────────────────┘     └────────────┬─────────────┘
                                     │ WSS (加密长连接)
                          ┌──────────┼──────────┐
                          │          │          │
                     ┌────┴───┐ ┌────┴───┐ ┌────┴───┐
                     │ Agent  │ │ Agent  │ │ Agent  │
                     │ 服务器A │ │ 服务器B │ │ 服务器C │
                     └────────┘ └────────┘ └────────┘

→ 适合: 不想自己维护 Server 的用户、需要团队协作的企业
→ 免部署，注册即用
→ Agent 仍然安装在用户自己的服务器上
```

#### 部署模式对比

| | 单机一体 (A) | 一主多从 (B) | 本机管远程 (C) | 云版 |
|--|-------------|-------------|--------------|------|
| Server 位置 | 目标服务器上 | 其中一台服务器 | 用户笔记本/PC | ServerPilot Cloud |
| Agent 位置 | 同一台机器 | 每台目标服务器 | 每台目标服务器 | 每台目标服务器 |
| 管理服务器数 | 1 台 | 2-20 台 | 1-10 台 | 无限 |
| 安装复杂度 | 一条命令 | 两步 | 两步 + 网络配置 | 注册即用 |
| 数据位置 | 本机 | 管理节点 | 本地电脑 | 云端 |
| 适合场景 | 个人开发者 | 小团队 | 远程运维 | 企业/团队 |
| 费用 | 免费 | 免费 | 免费 | $19/月起 |

| 横向扩展 | ✅ 加 Agent 只需装轻量守护进程 | ❌ 每台都装完整系统 |

### 4.4 断线重连与容错

网络中断是运维场景中最常见的问题之一。ServerPilot 在架构层面系统性处理断线和故障恢复。

#### Agent 断线重连

```
Agent 与 Server 的 WSS 连接断开时:

1. 检测断线 (心跳超时 90 秒无响应)
2. 指数退避重连:
   第 1 次: 立即重连
   第 2 次: 等待 2 秒
   第 3 次: 等待 4 秒
   第 4 次: 等待 8 秒
   ...
   最大间隔: 5 分钟
   持续重连直到成功 (永不放弃)

3. 重连成功后:
   · 发送完整环境快照 (确保 Server 端档案最新)
   · 上报断线期间积累的监控数据
   · 检查是否有未完成的任务

4. Agent 断线期间:
   · 已在执行的命令继续执行到完成
   · 执行结果缓存在本地队列，重连后批量上报
   · 定时任务正常执行 (不依赖 Server 在线)
   · 监控数据本地缓存 (最多保留 24 小时)
```

#### 任务执行中断处理

```
多步骤任务执行到第 N 步时出现中断:

场景 1: Agent 断线 (网络中断)
  · 当前步骤继续执行到完成
  · 后续步骤暂停，标记为 "pending-reconnect"
  · Agent 重连后，Server 询问用户: "继续还是回滚?"
  · 用户选择继续 → 从第 N+1 步恢复
  · 用户选择回滚 → 执行已记录的回滚命令

场景 2: Server 重启
  · 任务状态持久化在 SQLite 中，Server 重启后恢复
  · Agent 自动重连后，Server 检查未完成任务
  · 恢复策略同场景 1

场景 3: Agent 进程崩溃
  · systemd 自动重启 Agent (RestartSec=5)
  · Agent 启动时检查本地任务队列
  · 已执行的命令不会重复执行 (幂等性标记)

场景 4: AI Provider 不可用
  · 对话请求返回 "AI 暂时不可用"
  · 已在执行中的任务不受影响 (计划已生成)
  · 用户可切换到其他 Provider 或等待恢复
  · 紧急情况可使用预设模板命令 (不依赖 AI)
```

#### 消息可靠性

```
Server ↔ Agent 通信采用消息确认机制:

  Server → Agent: 下发命令 (附带 messageId)
  Agent → Server: 确认收到 (ack messageId)

  未收到 ack → Server 重发 (最多 3 次，间隔 5 秒)
  仍未收到 → 标记 Agent 为 "疑似离线"，Dashboard 告警

  Agent → Server: 上报结果 (附带 messageId)
  Server → Agent: 确认收到 (ack messageId)

  未收到 ack → Agent 本地队列缓存，重连后批量上报
```

### 4.5 Server 自身备份与容灾 *(新增)*

社区版 Server 为单节点架构，需要系统性保障 Server 自身的数据安全:

```
Server 数据备份策略:

1. SQLite 自动备份
   · 每日自动备份 SQLite 数据库到 /var/serverpilot/backups/
   · 备份方式: SQLite Online Backup API (不锁库，不影响服务)
   · 保留策略: 最近 7 个日备份 + 最近 4 个周备份
   · 备份文件命名: server-db-20260208-030000.sqlite.bak

2. 配置备份
   · Server 配置文件 (config.yaml) 每次修改后自动备份
   · Agent 密钥、证书文件一并备份
   · 用户 AI Provider 配置 (不含 API Key 明文)

3. 一键恢复
   · serverpilot restore --from /path/to/backup.sqlite.bak
   · 恢复内容: 服务器档案、操作历史、定时任务、告警规则
   · 不恢复: API Key (需重新配置)

4. Server 宕机时的 Agent 行为
   · Agent 进入离线自治模式 (参见 4.4 断线重连)
   · 已配置的定时任务正常执行 (Agent 本地缓存任务列表)
   · 监控数据本地缓存，Server 恢复后批量上报
   · 不接受新的 AI 对话任务 (需要 Server 在线)

5. 迁移与灾备
   · 支持导出完整 Server 数据: serverpilot export --output backup.tar.gz
   · 在新机器上导入: serverpilot import --from backup.tar.gz
   · 导入后 Agent 自动重连 (修改 DNS/IP 指向新 Server 即可)

6. 云版高可用 (商业版)
   · Server 多副本部署 (负载均衡)
   · 数据库 PostgreSQL 主从复制
   · 自动故障转移，99.9% SLA
```

### 4.6 WSS 通信安全与证书管理 *(新增)*

Server 与 Agent 之间的 WSS 连接必须加密，不同部署模式下的 TLS 证书获取方式不同:

```
证书管理策略 (按部署模式):

模式 A — 单机一体:
  · Server 和 Agent 在同一台机器上
  · 默认使用 localhost 回环连接，跳过 TLS (安全: 流量不出机器)
  · 如果用户配置了域名访问 Dashboard → 自动申请 Let's Encrypt 证书
  · 同一证书同时服务 Dashboard HTTPS 和 Agent WSS

模式 B — 一主多从 (内网):
  · 安装时自动生成自签名 CA + Server 证书
  · CA 证书指纹在 Agent 安装命令中携带:
    curl ... | bash -s -- --server wss://IP:3000 --token xxx \
      --ca-fingerprint sha256:abcdef...
  · Agent 首次连接时验证 Server 证书指纹 (TOFU: Trust On First Use)
  · 指纹匹配后建立信任，后续自动重连
  · 证书自动轮换: 到期前 30 天自动续签，Agent 平滑切换

模式 B — 一主多从 (公网):
  · 推荐用户为管理节点配置域名 + Let's Encrypt 证书
  · 安装脚本自动检测: 有域名 → 申请真证书; 无域名 → 自签名 + 指纹验证
  · 内置 ACME 客户端 (类似 Caddy 的自动证书管理)

模式 C — 本机管远程:
  · 与模式 B 公网场景相同
  · 额外支持: 通过内网穿透工具 (frp/ngrok) 暴露时的证书处理

云版:
  · ServerPilot Cloud 使用平台托管的 TLS 证书
  · Agent 内置 Cloud 根证书，直接验证
  · 用户无需任何证书配置

安全防护:
  · 证书指纹 Pinning: 防止中间人攻击
  · 不信任系统 CA 存储 (Agent 只信任自己的 CA 或指纹)
  · 证书过期自动告警 (Dashboard + 邮件)
  · 支持手动导入企业 CA 证书 (企业内网 PKI 场景)
```

### 4.8 服务器档案系统 (Server Profile)

每台服务器拥有一份持久化的独立档案，Agent 自动采集并增量更新，AI 对话时自动注入上下文。

#### 档案数据结构

```
ServerProfile {
  // === 基础身份 ===
  serverId: "srv-abc123"             // 唯一标识
  name: "prod-01"                    // 用户命名
  tags: ["production", "web"]        // 自定义标签
  addedAt: "2026-01-15"              // 接入时间

  // === 系统信息 (Agent 首次连接时采集，定期更新) ===
  os: "Ubuntu 22.04 LTS"

  cpu: { cores: 4, model: "Intel Xeon E5-2680" }
  memory: { total: "8GB" }
  disk: { total: "100GB", type: "SSD" }
  network: { publicIp: "1.2.3.4", privateIp: "10.0.0.5" }

  // === 软件清单 (Agent 每 5 分钟增量更新) ===
  installedSoftware: [
    { name: "nginx", version: "1.24", configPath: "/etc/nginx/", ports: [80, 443] },
    { name: "mysql", version: "8.0", dataDir: "/var/lib/mysql", configPath: "/etc/mysql/" },

    { name: "docker", version: "24.0", containers: 3 },

  runningServices: [
    { name: "nginx", status: "running", ports: [80, 443], uptime: "15d" },
    { name: "mysql", status: "running", ports: [3306], uptime: "15d" },
    { name: "myapp", status: "running", ports: [3000], manager: "pm2" },
  ]

  // === 历史操作摘要 (AI 自动总结，非原始日志) ===
  operationHistory: [
    "2026-01-15: 初始化服务器，安装 Nginx + MySQL + Node.js",
    "2026-01-20: 部署 myapp 项目到 /opt/myapp (Docker Compose)",
    "2026-02-01: 配置 SSL 证书 (Let's Encrypt, auto-renew)",
    "2026-02-05: 设置 MySQL 每日备份定时任务 → /backup",

  ]

  // === 已知注意事项 (AI 和用户共同维护) ===
  notes: [
    "MySQL root 密码存储在 /root/.my.cnf",
    "Nginx 配置使用自定义模板，勿用默认模板覆盖",
    "端口 8080 已被内部 API 网关占用",
    "每周日 03:00 有日志清理定时任务，此时段避免部署",
  ]

  preferences: {
    packageManager: "apt"            // 优先包管理器
    deploymentStyle: "docker"        // docker | bare-metal | pm2
    backupLocation: "/backup"        // 备份存放位置
    logLocation: "/var/log"          // 日志路径
    preferredEditor: "nano"          // 编辑器偏好
  }

```

#### 档案生命周期

```
Agent 首次连接
  → 全量采集系统信息、软件清单、运行服务
  → 创建初始档案

日常运行中
  → 心跳 (30s): 更新服务状态
  → 状态报告 (60s): 更新 CPU/内存/磁盘指标
  → 软件清单 (5min): 增量更新已安装软件
  → 操作完成后: AI 自动将操作总结追加到历史

用户对话时
  → Server 自动将该服务器档案注入 AI 上下文 (system prompt)
  → AI 基于档案生成精准的、个性化的操作方案

档案维护
  → 历史操作超过 50 条时，AI 自动合并归纳旧条目
  → 用户可手动添加/编辑 notes (已知注意事项)
  → 存储: 社区版用本地 SQLite，云版用云端数据库
```

#### 档案的 AI 上下文注入

每次 AI 对话时，服务器档案被注入为 system prompt 的一部分:

```
[System] 你正在管理以下服务器:

服务器: prod-01 (Ubuntu 22.04, 4核 8GB, 100GB SSD)
已安装: Nginx 1.24 (:80,:443), MySQL 8.0 (:3306), Node.js 22.1, Docker 24.0
```

#### 对话上下文管理 *(新增)*

AI 对话的上下文窗口 (context window) 有限，需要系统性管理:

```
上下文构成 (每次 AI 调用):

  ┌─ System Prompt ──────────────────────────────────┐
  │  角色定义 (~200 tokens)                            │
  │  安全规则 (~300 tokens)                            │
  │  输出格式约束 (~200 tokens)                         │
  │  服务器档案 (~800-2000 tokens, 压缩后)              │
  │  知识库检索结果 (~500-1500 tokens, 按需注入)        │
  ├─────────────────────────────────────────────────┤
  │  对话历史 (动态管理)                                │
  │  最近 N 轮对话 (~500-3000 tokens)                  │
  ├─────────────────────────────────────────────────┤
  │  当前用户消息 (~100-500 tokens)                     │
  └─────────────────────────────────────────────────┘
  总计: 约 2500-8000 tokens/次

上下文管理策略:

1. 滑动窗口
   · 保留最近 10 轮对话
   · 超过 10 轮时，丢弃最早的轮次
   · 关键操作结果始终保留 (如: 上一步安装了什么)

2. 对话摘要 (长会话场景)
   · 超过 20 轮时触发自动摘要
   · 用 AI 将前 15 轮压缩为一段摘要 (~300 tokens)
   · 摘要 + 最近 5 轮 = 新的上下文
   · 摘要示例: "用户在 prod-01 上安装了 Nginx 和 Redis，
     配置了反向代理到 3000 端口，SSL 证书已申请"

3. 多步操作的上下文传递
   · 多步骤任务执行时，每步结果自动追加到上下文
   · 步骤结果格式化为简洁摘要 (非原始输出)
   · 示例: "步骤 1 完成: apt update 成功" (而非完整 apt 输出)

4. 模型适配
   · 不同模型上下文窗口大小不同:
     Claude Sonnet: 200K tokens (充裕)
     GPT-4o: 128K tokens (充裕)
     Ollama 本地模型: 4K-32K tokens (需要激进压缩)
   · 小模型自动启用更激进的上下文压缩:
     - 服务器档案只保留关键字段
     - 对话历史只保留最近 5 轮
     - 知识库注入限制在 500 tokens 以内
```

### 6.8 Agent (客户端) 功能

Agent 是一个轻量级守护进程，安装在目标服务器上。**Agent 代码 100% 开源。**

#### 核心职责

```
Agent 不做决策，只做执行和汇报:

1. 系统监控与档案采集
   - 定期采集 CPU/内存/磁盘/网络指标
   - 发现已安装软件和运行服务
   - 探测开放端口
   - 增量更新服务器档案

2. 命令执行
   - 接收服务端下发的命令
   - 多层安全验证 (详见 安全架构)
   - 实时流式回传输出

3. 状态汇报
   - 心跳保活 (30秒)
   - 系统状态报告 (60秒)
   - 软件清单报告 (5分钟)

4. 本地安全保障
   - 命令分级白名单
   - 参数安全审计
   - 本地不可篡改操作日志
   - 操作前自动快照

5. 自动更新
   - 检查新版本
   - 安全下载并热替换
   - 二进制签名验证 (详见下方)
```

#### Agent 自动更新与签名验证 *(新增)*

```
签名方案: Ed25519 数字签名

1. 构建签名流程 (CI/CD):
   · GitHub Actions 构建 Agent 二进制
   · 使用项目维护者的 Ed25519 私钥对二进制签名
   · 私钥存储在 GitHub Secrets 中 (不入库)
   · 发布产物: agent-linux-amd64 + agent-linux-amd64.sig

2. Agent 验证流程:
   · Agent 内置 Ed25519 公钥 (硬编码在二进制中)
   · 下载新版二进制 + 签名文件
   · 用内置公钥验证签名 → 不匹配则拒绝更新
   · 验证通过 → 热替换 (先写临时文件，rename 原子操作)
   · systemd 自动重启新版 Agent

3. 公钥轮换:
   · 新版 Agent 可携带下一代公钥 (双公钥过渡期)
   · 当前版本同时信任新旧两个公钥
   · 旧公钥在 2 个大版本后移除

4. 下载安全:
   · 下载源: GitHub Releases (HTTPS)
   · 备用源: 自建 CDN (可配置)
   · 下载超时: 30 秒
   · 文件完整性: SHA-256 校验 + Ed25519 签名双重验证
   · 防回滚: 新版本号必须大于当前版本 (防降级攻击)

5. 更新策略:
   · 自动检查: 每 6 小时检查一次
   · 用户可选: 自动更新 / 提示更新 / 禁用更新
   · 更新窗口: 避免在任务执行期间更新
   · 回滚: 保留上一版二进制，新版启动失败自动回滚
```

#### 安装要求

- 支持: Ubuntu 18+, Debian 10+, CentOS 7+, RHEL 8+, macOS 12+
- 资源占用: < 50MB 内存, < 1% CPU
- 无需额外依赖 (单一二进制文件)
- 支持 systemd 管理 (Linux)
- 支持 launchd 管理 (macOS)
- **Windows Server**: 当前不支持。原因: 目标用户以 Linux 服务器为主 (宝塔用户群体 99% 为 Linux)。Windows 支持列入 v1.3+ 长期规划

## 7. 安全架构

安全是 ServerPilot 的核心竞争力。用户最大的顾虑是"AI 会不会删库跑路？"。我们通过**五层纵深防御**确保安全。

### 7.1 设计原则

```
1. Agent 永远不直接接触 AI — 只接收和执行经过 Server 审核的具体命令
2. 默认只读 — 新连接的服务器默认只有只读权限，用户主动提升
3. 透明可见 — 每个命令执行前完整展示，不隐藏任何操作
4. 可回滚 — 任何修改操作之前，自动创建可恢复的快照
5. 开源可审计 — Agent 代码 100% 开源，用户可自行审查
```

### 7.2 Agent 运行权限模型 *(新增)*

Agent 进程的系统权限直接决定安全上限。采用**专用用户 + sudoers 白名单**的最小权限方案:

```
Agent 进程权限设计:

1. 专用系统用户
   · 安装时自动创建 serverpilot 用户和同名用户组
   · Agent 以 serverpilot 用户身份运行 (非 root)
   · 家目录: /var/serverpilot/ (存放配置、日志、快照)

2. sudoers 分级授权
   · 安装时自动写入 /etc/sudoers.d/serverpilot:

   # 🟢 只读级: 无需 sudo
   # ls, cat, df, free, top 等 — serverpilot 用户直接执行

   # 🟡 安装级: sudo 免密码 (预定义命令)
   serverpilot ALL=(root) NOPASSWD: /usr/bin/apt update, /usr/bin/apt install *
   serverpilot ALL=(root) NOPASSWD: /usr/bin/systemctl status *

   # 🔴 修改级: sudo 需要 Server 下发的一次性令牌验证
   serverpilot ALL=(root) NOPASSWD: /usr/bin/systemctl restart *
   serverpilot ALL=(root) NOPASSWD: /usr/bin/systemctl stop *

   # ⛔ 破坏级: sudo + 用户二次确认 + 自动快照
   # (通过 Agent 内部逻辑控制，非 sudoers 层面)

3. 文件系统权限
   · Agent 只有 /var/serverpilot/ 目录的写权限
   · 修改系统配置文件 (/etc/*) 必须通过 sudo
   · 快照备份目录: /var/serverpilot/snapshots/ (仅 serverpilot 可写)
   · 审计日志: /var/serverpilot/audit.log (append-only, root 才能删除)

4. 网络权限
   · Agent 仅允许向配置的 Server 地址发起 WSS 出站连接
   · 不监听任何端口 (无入站连接)
   · 可选: 通过 iptables/nftables 规则限制 Agent 网络访问范围

5. 容器化隔离 (可选增强)
   · 支持以 Docker 容器方式运行 Agent
   · 挂载宿主机必要目录 (只读 + 按需写入)
   · 通过 Linux capabilities 进一步限制权限
   · 适合安全要求极高的生产环境
```

**为什么不直接以 root 运行:**

| 方案 | 便利性 | 安全性 | 选择理由 |
|------|--------|--------|---------|
| root 运行 | 高 (无需 sudoers) | 低 (进程被利用 = 完全控制) | ❌ |
| 专用用户 + sudoers | 中 | 高 (权限最小化) | ✅ 推荐 |
| 容器化运行 | 低 (配置复杂) | 极高 (内核级隔离) | ✅ 可选增强 |

### 7.3 第一层: 命令分级制度 (Agent 端)

所有命令按风险等级分为五个级别，每个级别有不同的执行策略:

```
🟢 只读级 (自动执行，无需确认):
   ls, cat, head, tail, df, free, top, ps, systemctl status,
   uname, uptime, whoami, id, hostname, date, which, type,
   docker ps, docker images, nginx -t

🟡 安装级 (需用户确认):
   apt install, apt update, npm install, pip install,
   docker pull, brew install, yum install,
   git clone, curl (下载), wget

🔴 修改级 (需确认 + 显示影响范围):
   systemctl restart/stop/start, service restart,
   nginx -s reload, 修改配置文件,
   docker stop, docker restart,
   crontab -e, chmod, chown

⛔ 破坏级 (需确认 + 二次密码验证 + 操作前快照):
   rm (非临时文件), apt remove/purge, docker rm,
   DROP DATABASE, DROP TABLE, truncate,
   userdel, groupdel

🚫 绝对禁止 (Agent 拒绝执行，无例外):
   rm -rf /
   mkfs, fdisk, dd if=/dev/zero
   :(){ :|:& };:  (fork bomb)
   > /dev/sda
   chmod -R 777 /
   任何指向系统根目录的递归删除
```

### 7.4 第二层: 参数安全审计 (Agent 端)

白名单只验证命令名不够，还需要验证命令参数:

```
危险参数黑名单:
  --purge, --force, --no-preserve-root, -rf (组合)

危险路径保护 (这些路径下的破坏性操作需要额外确认):
  /etc, /boot, /usr, /var/lib/mysql, /var/lib/postgresql,
  /home (非当前用户目录)

命令语义分析:
  AI 生成计划时标注每个命令的风险等级
  → Agent 端二次验证，如果 Agent 判定的风险高于 AI 标注的，以 Agent 为准
```

### 7.5 第三层: 操作前快照与回滚 (Agent 端)

```
所有 🔴 及以上风险的操作，自动执行:

1. 快照创建
   - 配置文件: 自动备份到 /var/serverpilot/snapshots/<timestamp>/
   - 数据库: 执行 mysqldump / pg_dump 轻量快照
   - 服务状态: 记录当前运行状态

2. 回滚能力
   - Dashboard 一键回滚到快照点
   - AI 对话回滚: "回滚 prod-01 到上次操作之前的状态"
   - 自动回滚: 操作超时或异常退出时自动触发

3. 快照保留策略
   - 保留最近 20 个快照
   - 超过 30 天的快照自动清理
   - 重要快照可标记为"永久保留"

4. 快照覆盖范围与边界 (新增)
   ✅ 可回滚:
   - 配置文件修改 (/etc/nginx/, /etc/mysql/ 等)
   - 数据库数据 (mysqldump / pg_dump 快照)
   - systemd service 文件
   - crontab 定时任务
   - 用户指定的自定义文件/目录

   ⚠️ 部分可回滚:
   - 软件包安装 (apt install): 记录安装了哪些包，
     回滚时执行 apt remove，但可能有残留依赖
   - Docker 容器: 记录操作前的镜像和容器状态，
     回滚时恢复，但新拉取的镜像不自动删除

   ❌ 不可回滚 (明确告知用户):
   - 内核升级 (需要重启才能生效/回退)
   - 已发送的网络请求 (如: 已调用的外部 API)
   - 已清空的日志文件 (truncate 操作)
   - 硬件相关操作 (磁盘分区等)

   AI 生成计划时会标注每个步骤的可回滚性:
     [可回滚] 修改 Nginx 配置
     [可回滚] 重启 Nginx 服务
     [不可回滚] 清理旧日志文件 — 请确认后再执行
```

**示例流程:**

```
用户: "帮我升级 MySQL 到 8.0"

AI 计划:
  1. [自动] 创建 MySQL 数据快照 → /var/serverpilot/snapshots/20260208-153000/
  2. [自动] 记录当前 MySQL 版本和配置
  3. [需确认] 执行升级: apt install mysql-server-8.0
  4. [自动] 验证升级结果: mysql --version
  5. [自动] 保留回滚脚本 (如果失败可一键恢复)

  风险等级: ⛔ 破坏级 (数据库升级)
  需要输入密码确认。
```

### 7.6 第四层: 紧急制动 (Server 端 + Dashboard)

```
Kill Switch:
  - Dashboard 红色 "紧急停止" 按钮，一键终止所有正在执行的操作
  - Agent 收到 kill 信号后立即终止当前进程
  - 支持按服务器粒度停止

操作超时:
  - 每个命令有预设超时时间
  - 超过预期时间 2 倍自动终止并告警

异常行为检测:
  - 短时间内大量文件删除 → 自动暂停 + 告警
  - 磁盘写入量异常 → 自动暂停 + 告警
  - 未知进程启动 → 告警
```

### 7.7 第五层: 审计与可追溯 (全链路)

```
不可篡改日志:
  - Agent 本地: /var/serverpilot/audit.log (append-only)
  - Server 端: 数据库审计表 (云版)
  - 每条记录: 时间 + 执行者 + 命令 + 输出 + 风险等级 + AI 对话上下文

操作回放:
  - 完整重现历史操作的命令和输出
  - 支持导出为报告 (云版企业版)
```

## 8. AI 质量与可靠性

AI 是 ServerPilot 的核心引擎，但 AI 模型天生具有不确定性 — 可能生成不存在的命令、错误的参数、甚至幻觉出虚假的包名。我们必须在产品层面系统性地解决这个问题。

### 8.1 问题: 不同模型能力差异巨大

```
模型能力光谱:

  Claude Opus 4 ──── GPT-4o ──── DeepSeek ──── Qwen 14B ──── Llama 8B
  (最强)                                                      (最弱)

  同一个需求 "在 Ubuntu 上安装 Nginx 并配置 SSL":
  · Claude: 完整的 apt install + certbot + nginx 配置，考虑 UFW 防火墙
  · Llama 8B: 可能只给 apt install nginx，遗漏 SSL 步骤或给出过时命令
```

**挑战:** 用户可能用最便宜/本地的小模型，我们不能假设 AI 一定正确。

### 8.2 三层 AI 质量防线

```
┌─────────────────────────────────────────────────┐
│  第一层: Prompt 工程 — 让 AI 尽量生成正确的命令     │
├─────────────────────────────────────────────────┤
│  第二层: 输出校验 — 执行前验证命令合法性             │
├─────────────────────────────────────────────────┤
│  第三层: 执行反馈 — 失败后自动诊断和修复             │
└─────────────────────────────────────────────────┘
```

#### 第一层: System Prompt 工程

每次 AI 对话的 System Prompt 由四部分动态拼接:

```
[1. 角色定义]
你是 ServerPilot AI 运维助手。你的任务是生成安全、准确的 Linux 运维命令。
你必须严格遵循以下规则...

[2. 服务器档案] (自动注入)
服务器: prod-01 (Ubuntu 22.04, 4核 8GB)
已安装: Nginx 1.24, MySQL 8.0, Docker 24.0...
(详见 4.8 服务器档案)

[3. 安全规则] (硬编码)
· 绝不生成 rm -rf /、mkfs、dd 等破坏性命令
· 所有修改操作必须标注风险等级
· 涉及数据的操作必须先备份
· 不确定的命令必须标注 [需验证]

[4. 输出格式约束] (结构化)
必须返回 JSON 格式的执行计划:
{
  "steps": [
    {
      "command": "apt update",
      "risk": "green",
      "description": "更新软件源",
      "rollback": null
    }
  ],
  "totalRisk": "yellow",
  "estimatedTime": 120
}
```

**关键设计:** 要求 AI 返回**结构化 JSON** 而非自由文本，这样 Server 端可以精确解析和校验每一条命令。

#### 第二层: 命令输出校验 (Server 端，执行前)

```
AI 生成命令 → Server 端校验层 → 通过 → 发送给 Agent 执行
                    │
                    ├─ 1. 格式校验: JSON 是否合法、字段是否完整
                    │
                    ├─ 2. 命令存在性校验:
                    │     · 命令名是否在已知命令库中
                    │     · 包名是否在包管理器索引中 (apt search / npm search)
                    │     · 文件路径是否合理 (不指向 /dev, /proc 等)
                    │
                    ├─ 3. 安全规则校验:
                    │     · 命令风险等级是否与 AI 标注一致
                    │     · 是否触发绝对禁止列表
                    │     · 参数黑名单检查
                    │
                    ├─ 4. 上下文一致性:
                    │     · 命令是否与目标 OS 匹配 (不在 Ubuntu 上用 yum)
                    │     · 软件版本是否与档案中一致
                    │
                    └─ 不通过 → 要求 AI 重新生成 (最多重试 2 次)
                              → 仍失败 → 回退到预设模板命令
```

#### 第三层: 执行反馈闭环

```
命令执行 → exitCode != 0?
             │
             ├─ 是 → 将 stderr 发回 AI，请求诊断
             │       AI 分析错误原因，生成修复命令
             │       修复命令再次经过第二层校验
             │       最多自动重试 1 次，超过则暂停请求用户决策
             │
             └─ 否 → 验证预期输出
                     · 安装命令: 检查 --version 是否有输出
                     · 配置命令: 检查配置语法 (nginx -t, mysql --check)
                     · 服务命令: 检查 systemctl status
                     · 验证不通过 → 同上，AI 诊断 + 修复
```

### 12.6 版本升级与数据迁移策略

#### 自动升级机制

```
Server 升级 (Docker Compose 环境):
  1. 拉取新镜像: docker compose pull
  2. 重启服务: docker compose up -d
  3. 自动运行数据库迁移 (SQLite migrate)
  4. 验证健康检查通过
  5. 失败时自动回滚到上一版镜像

Agent 升级:
  1. Agent 每次启动时检查 Server 版本
  2. Server 通知 Agent 有新版本可用
  3. Agent 下载新版二进制 → 校验签名 → 热替换
  4. systemd 自动重启新版 Agent
  5. 支持回滚: 保留上一版二进制
```

#### 数据库 Schema 迁移

```
SQLite 迁移策略:
  · 每个版本附带迁移脚本 (forward-only)
  · Server 启动时自动检测当前 schema 版本
  · 按顺序执行所有未应用的迁移
  · 迁移前自动备份 SQLite 文件
  · 迁移失败 → 恢复备份 → 阻止启动 → 告警

  示例:
  migrations/
    001_init.sql
    002_add_server_profile_preferences.sql
    003_add_scheduled_tasks.sql
    ...
```

#### 社区版 → 云版数据迁移

```
用户从自部署切换到云版时:
  1. Dashboard 提供 "导出数据" 功能
     · 导出格式: JSON (服务器档案 + 操作历史 + 定时任务 + 设置)
     · 不含敏感数据 (密码、API Key)
  2. 云版注册后，上传导出文件
  3. 自动导入所有数据
  4. Agent 修改连接地址到云端 Server
     serverpilot-agent --server wss://api.serverpilot.dev
```

#### 版本兼容性

```
Agent 与 Server 版本兼容策略:
  · Server 向后兼容至少 2 个 Agent 小版本
  · Agent 与 Server 版本不一致时:
    - 差 1 个小版本: 正常运行，Dashboard 提示升级
    - 差 2 个小版本: 功能降级，强提示升级
    - 差 1 个大版本: 拒绝连接，要求升级
  · WSS 协议版本号在握手时交换
```

## 13. 通信协议与扩展

### 13.1 WSS 通信协议规范

Server 与 Agent 之间通过 WebSocket (WSS) 通信，消息格式为 JSON。

#### 消息格式

```json
{
  "id": "msg-uuid-v4",           // 消息唯一 ID (用于 ack)
  "type": "command.execute",      // 消息类型
  "timestamp": 1707350400000,     // Unix 毫秒时间戳
  "payload": { ... }              // 消息体 (各类型不同)
}
```

#### 核心消息类型

```
Server → Agent:
  auth.challenge          握手认证挑战
  command.execute         执行命令
  command.cancel          取消执行中的命令
  env.request             请求环境信息
  agent.upgrade           通知升级

Agent → Server:
  auth.response           认证响应
  env.report              环境信息上报
  command.output          命令实时输出 (流式)
  command.result          命令执行结果
  heartbeat               心跳
  metrics.report          监控指标上报

双向:
  ack                     消息确认
  error                   错误通知
```

#### 认证握手流程

```
Agent                              Server
  │                                  │
  ├── WSS 连接 ──────────────────────►│
  │                                  │
  │◄──── auth.challenge (nonce) ──────┤
  │                                  │
  ├── auth.response ─────────────────►│
  │    (签名 = HMAC(nonce, 持久密钥))  │
  │                                  │
  │◄──── auth.success ────────────────┤
  │      (protocol_version, config)   │
  │                                  │
  ├── env.report (完整环境快照) ──────►│
  │                                  │
  │    连接建立，开始正常通信            │
```

### 13.2 知识库累积体系

AI 模型对 Nginx、MySQL、Docker 等主流软件了如指掌，但面对小众软件、新发布的项目、或者企业内部工具时，AI 的知识就可能过时或缺失。ServerPilot 的知识库是一个**自我成长的知识系统** — 用得越多、积累越多、AI 越准。

#### 四层知识来源

```
┌─────────────────────────────────────────────────────────────┐
│                    AI 对话时的知识检索                          │
│                                                             │
│  用户: "帮我安装 xxx"                                         │
│       ↓                                                     │
│  ① 内置知识库 ── 有 → 直接注入 AI 上下文                      │
│       ↓ 没有                                                │
│  ② 用户本地知识库 ── 有 → 注入 (之前成功装过/手动添加的)       │
│       ↓ 没有                                                │
│  ③ 社区共享知识库 ── 有 → 注入 (其他用户贡献的经验)            │
│       ↓ 没有                                                │
│  ④ 自动抓取 ── 去 GitHub/官网抓取文档 → 解析 → 注入 → 缓存    │
│       ↓ 都没有                                              │
│  AI 凭通用知识尝试 (标注 [未验证，请检查])                      │
└─────────────────────────────────────────────────────────────┘
```

**③ 文档自动抓取: 遇到未知软件时**

```
用户: "帮我安装 Deno" (知识库里没有 Deno)

Server 自动抓取流程:
  1. 检测到知识库中无 "deno" 相关知识
  2. 触发文档抓取:
     a. 搜索 GitHub: "deno" → 找到 denoland/deno
     b. 抓取 README.md → 提取安装命令
     c. 抓取官方文档 (deno.land) → 提取安装指南
     d. 解析安装步骤、系统要求、验证方法
  3. 解析结果缓存到本地知识库:
     knowledge-base/
       └─ _auto/              # 自动抓取的知识 (区别于内置)
           └─ deno/
               ├─ github-readme.md
               └─ official-docs.md
  4. 将抓取内容注入 AI 上下文
  5. AI 结合官方文档生成安装计划

抓取策略:
  · GitHub README: 最优先 (markdown 格式，易解析)
  · 官方文档: 搜索 "{software} installation guide"
  · 包管理器: npm/pip/brew info
  · 缓存策略: 7 天有效期，过期重新抓取
  · 速率限制: 每分钟最多 5 次抓取请求

技术实现:
  · 抓取: fetch + cheerio/mozilla-readability (HTML → Markdown)
  · 解析: AI 辅助提取安装步骤 (一次性 prompt)
  · 存储: 本地文件系统 (knowledge-base/_auto/)
  · 检索: 复用已有的向量搜索引擎 (TF-IDF + 余弦相似度)
```

```yaml
  - description: "启用站点"

    risk: yellow
  - description: "检查配置语法"
    command: "nginx -t"
    risk: green
  - description: "重载 Nginx"
    command: "systemctl reload nginx"

verify:
  command: "curl -s -o /dev/null -w '%{http_code}' http://localhost"
```

```
  · 监控插件: Prometheus exporter, Grafana dashboard
  · 部署插件: Docker Compose 一键部署应用模板
  · 云平台插件: AWS/阿里云/腾讯云 API 集成

MVP 阶段不做插件系统，优先做好知识库累积。
插件系统在社区规模达到一定程度后再设计 (避免过早抽象)。
```

### 13.3 数据隐私与合规

#### 社区版 (数据完全自主)

```
· 所有数据存储在用户自己的 Server 上 (SQLite)
· 不向任何第三方发送数据
· AI API 调用直接从 Server 到用户选择的 AI Provider
  (数据流: Server → Claude/OpenAI API，不经过 ServerPilot)
· Agent 只与用户指定的 Server 通信
· 无遥测、无埋点、无 phone-home
  (可选: 匿名安装统计，默认关闭，用户主动 opt-in)
```

#### 云版 (最小化数据收集)

```
我们收集的数据:
  · 用户账户信息 (邮箱、密码哈希)
  · 服务器档案 (系统信息、软件清单)
  · 操作历史记录
  · AI 对话记录 (用于改进服务)

我们不收集的数据:
  · 服务器上的文件内容
  · 数据库内容
  · 用户的 SSH 密钥或密码
  · 服务器的网络流量

数据存储:
  · 数据库: 加密存储 (AES-256)
  · 传输: 全链路 TLS 加密
  · 位置: 用户可选数据中心区域

用户权利:
  · 随时导出全部数据 (JSON 格式)
  · 随时删除账户和所有关联数据
  · 30 天内数据彻底清除

合规:
  · GDPR 就绪 (数据导出、删除权、数据处理说明)
  · 中国数据安全法: 云版中国区独立部署 (数据不出境)
```

#### 云版多租户隔离架构 *(新增)*

```
云版 Server 服务多个租户 (团队)，需要严格的数据隔离:

1. 数据库隔离
   · 采用 Schema 级别隔离 (非共享表):
     每个租户一个独立的 PostgreSQL Schema
     tenant_abc123.servers, tenant_abc123.operations, ...
   · 所有数据库查询自动注入 tenant_id 过滤条件 (ORM 层拦截)
   · 防止 SQL 注入导致的跨租户数据泄露

2. Agent 连接隔离
   · 每个 Agent 只属于一个租户
   · Agent 认证时携带 tenant_id + agent_key
   · Server 端 WSS 连接管理器按租户分组
   · 一个租户的 Agent 永远无法接收到另一个租户的命令

3. AI 对话隔离
   · 每个租户的 AI 对话上下文完全独立
   · 服务器档案、知识库检索仅在租户范围内
   · 不同租户的对话记录不会交叉

4. 资源限制 (防止单租户占用过多资源)
   · 并发 AI 对话数: 专业版 5 / 企业版 无限
   · 并发 Agent 连接数: 按套餐配额
   · API 请求速率: 100 req/min (专业版) / 1000 req/min (企业版)

5. 安全审计
   · 管理员操作日志与普通用户操作日志分离
   · 跨租户访问尝试自动告警 (安全事件)
   · 定期自动化渗透测试 (租户隔离有效性验证)
```

## 附录 C: 开源协议选择论证

开源协议的选择直接影响产品的商业保护和社区活力，一旦发布很难更改，需要慎重。

### 候选协议对比

| 协议 | 代表项目 | 社区友好度 | 商业保护 | 风险 |
|------|---------|-----------|---------|------|
| **MIT** | Vite, Next.js | 极高 | 无 | 竞争对手可直接 fork 做闭源商业版 |
| **Apache 2.0** | Supabase, K8s | 高 | 低 (专利保护) | 同上，但有专利防御 |
| **AGPL 3.0** | Grafana, Minio | 中 | 高 | 企业用户可能避讳 AGPL |
| **BSL 1.1** | HashiCorp, CockroachDB | 中低 | 极高 | 社区争议大，不算"真开源" |
| **SSPL** | MongoDB, Elastic | 低 | 极高 | OSI 不认可，社区反感 |
| **ELv2 (Elastic)** | Elasticsearch | 中低 | 高 | 较新，认知度低 |

### 推荐方案: 分层协议

```
Agent (客户端代码)   → Apache 2.0
  · 安装在用户服务器上，必须最大化信任
  · Apache 2.0 有专利保护，比 MIT 更安全
  · 企业用户接受度最高

Server (服务端代码)  → AGPL 3.0 或 BSL
  · 核心商业价值在 Server (AI 引擎、Dashboard、调度)
  · AGPL: 任何修改后的 Server 代码必须开源
    → 竞争对手不能 fork 做闭源 SaaS
    → 但代码完全透明，用户信任度高
  · BSL: 源码可见，但商业使用需要许可证
    → 保护更强，但"开源"标签有争议

知识库/模板         → MIT
  · 社区贡献门槛最低
  · 允许任何人自由使用
```

### 建议: Agent Apache 2.0 + Server AGPL 3.0

```
理由:
1. Agent Apache 2.0 = 最大化用户信任 (装在生产服务器上的代码)
2. Server AGPL 3.0 = 保护商业模式 (竞争对手不能 fork 做闭源 SaaS)
3. 代码完全透明 = 区别于宝塔的核心卖点
4. 参考先例: Grafana (AGPL), Minio (AGPL) 都成功商业化
5. 避免 BSL 的"伪开源"争议，在开源社区获得更好的口碑

风险缓解:
· AGPL 对自部署用户无影响 (不对外提供服务)
· 只有想用 Server 代码做 SaaS 的竞争对手才受限
· 这正是我们想要的保护
```
