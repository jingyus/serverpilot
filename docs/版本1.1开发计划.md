## 零、v1.0 遗留项：核心 AI 功能实现（优先级：P0）

> **状态**：当前前后端通信已打通，但服务端返回空安装计划。需要实现真实的 AI 功能。

### 0.1 服务端 AI Agent 核心逻辑

#### 0.1.1 安装计划生成器（AI Planner）

当前 `handleEnvReport` 返回空计划，需要实现：

- [ ] **环境分析 AI 调用**：
  - 调用 Claude API 分析环境信息（`EnvironmentInfo`）
  - Prompt 工程：基于 OS、Shell、运行时、包管理器生成定制化建议
  - 输出：适合该环境的安装方案描述

- [ ] **安装计划生成**：
  - 根据 AI 分析结果 + 软件知识库（如 `installers/openclaw/`）
  - 生成具体的 `InstallPlan`（步骤列表、超时时间、回滚策略）
  - 每个步骤包含：`id`, `description`, `command`, `timeout`, `onError`

- [ ] **知识库集成**：
  - 从 `knowledge-base/openclaw/` 加载安装文档
  - 将文档内容嵌入 AI prompt，提供安装上下文
  - 支持 Markdown 格式的知识库文件

- [ ] **环境适配器**：
  - macOS + Homebrew → 使用 `brew install` 命令
  - Linux + apt → 使用 `apt-get install` 命令
  - 根据已安装的包管理器自动选择最佳方案

**实现位置**：
- `packages/server/src/ai/planner.ts` — 计划生成主逻辑
- `packages/server/src/installers/openclaw/planner.ts` — OpenClaw 专用计划（已有，需完善）
- `packages/server/src/knowledge/loader.ts` — 知识库加载器

#### 0.1.2 错误诊断与修复建议（AI Error Analyzer）

当前 `handleErrorOccurred` 未实现，需要：

- [ ] **错误上下文分析**：
  - 接收 `ErrorContext`（命令、退出码、stdout、stderr、环境、历史步骤）
  - 提取关键错误信息（错误代码、依赖缺失、权限问题等）

- [ ] **AI 诊断调用**：
  - Prompt 工程：将错误上下文结构化描述给 AI
  - 要求 AI 分析失败原因并给出修复方案
  - 流式输出诊断过程（`ai.stream.*` 消息）

- [ ] **修复策略生成**：
  - AI 返回 1-3 个 `FixStrategy`（修复方案）
  - 每个方案包含：命令列表、置信度、风险等级、是否需要 sudo
  - 按置信度排序，优先展示最可能成功的方案

- [ ] **常见错误规则库**：
  - 预定义常见错误模式（如 `command not found`, `permission denied`）
  - 优先匹配规则库，命中则跳过 AI 调用（降成本）
  - 规则库未覆盖时再调用 AI

**实现位置**：
- `packages/server/src/ai/error-analyzer.ts` — AI 错误分析（已有框架，需完善）
- `packages/server/src/ai/common-errors.ts` — 常见错误规则库（新增）
- `packages/server/src/api/handlers.ts` — `handleErrorOccurred` 完善

#### 0.1.3 AI Prompt 模板系统

当前 prompts 硬编码在代码中，需要优化：

- [ ] **模板化管理**：
  - 创建 `prompts/` 目录，分场景存放 prompt 模板
  - 场景：`env-analysis.md`, `plan-generation.md`, `error-diagnosis.md`
  - 支持 Mustache/Handlebars 模板变量替换

- [ ] **版本控制**：
  - Prompt 模板 Git 版本化
  - 方便 A/B 测试不同 prompt 的效果

- [ ] **动态加载**：
  - 服务启动时加载所有 prompt 模板到内存
  - 开发模式支持热重载（修改 prompt 无需重启）

**实现位置**：
- `packages/server/prompts/` — Prompt 模板目录（新增）
- `packages/server/src/ai/prompt-loader.ts` — 模板加载器（新增）

#### 0.1.4 知识库向量化（可选，后续优化）

当前知识库直接嵌入 prompt，token 消耗大。可优化为：

- [ ] 向量数据库集成（Pinecone / Qdrant / 本地 Chroma）
- [ ] 安装文档向量化索引
- [ ] RAG（检索增强生成）：根据环境信息检索相关文档片段
- [ ] 只将相关片段嵌入 prompt，减少 token 消耗 70%+

> **优先级**：P2，先用完整文档嵌入跑通流程，用户量大后再优化

## 二、安全与防滥用（优先级：高）

### 2.1 当前风险分析

| 风险 | 严重程度 | 说明 |
|------|----------|------|
| 无认证机制 | **高** | 任何人都可以连接 WebSocket，无需身份验证 |
| 伪造错误刷 AI | **高** | 客户端可反复发送 `error.occurred` 触发 AI 诊断，变相免费用 AI |
| 无频率限制 | **高** | 没有 rate limiting，单个连接可无限触发 AI 调用 |
| AI 流式响应直传 | **中** | `ai.stream.token` 把 AI 原始输出推给客户端，可被截获利用 |
| 协议无校验 | **中** | 客户端可发送任意 payload，服务端缺少深度校验 |

> **现状**：客户端目前没有自由对话功能（只有确认/选择/文本输入），暂不存在通过对话滥用 AI 的风险。但 `error.occurred` → `fix.suggest` 是最大滥用入口。

### 2.2 认证与鉴权（分阶段混合认证）

采用「机器码 + 激活码」混合方案，免费用户零摩擦，付费用户一次激活：

```
免费用户 ──→ 机器码（自动生成，用户无感知）
                │
                │ 额度用完 or 想用高级功能
                ▼
付费用户 ──→ 激活码（一次性输入，绑定机器）
                │
                │ 续费 / 换设备 / 管理账单
                ▼
           网页端（微信/支付宝付款，生成激活码）
```

#### 第一层：机器码（免费用户，自动无感知）
- [ ] 客户端自动生成设备指纹：`hash(hostname + os + arch + mac地址 + 用户名)`
- [ ] 首次连接时服务端自动分配 `device_token`
- [ ] 本地存储 `~/.aiinstaller/device.json`：
  ```json
  { "deviceId": "a1b2c3...", "token": "tok_xxxx", "createdAt": "2026-02-07" }
  ```
- [ ] 服务端按 deviceId 做免费限流：5 次安装/月

#### 第二层：激活码（付费用户，一次输入）
- [ ] 用户在网页端付款后获得激活码
- [ ] CLI 中一次性输入：`./install-agent activate XXXX-XXXX-XXXX`
- [ ] 激活后写入本地：
  ```json
  { "deviceId": "a1b2c3...", "token": "tok_xxxx",
    "licenseKey": "XXXX-XXXX-XXXX", "plan": "pro", "expiresAt": "2027-02-07" }
  ```
- [ ] 激活码默认绑定 1 台设备，专业版 3 台，企业版不限
- [ ] 换设备：网页端「解绑旧设备」→ 新设备重新输入激活码
- [ ] 续费在网页端操作，服务端自动延期，客户端无感知

#### 第三层：服务端统一校验（兜底，不信任客户端）
- [ ] WebSocket 握手时强制校验：
  - 无 token → 拒绝连接
  - device_token → 免费限流（5 次/月，每次最多 20 次 AI 调用）
  - license_key → 按付费套餐放行
- [ ] 每次 AI 调用前检查额度，超限返回升级引导
- [ ] 会话级权限控制：免费/付费/企业，不同级别调用不同 AI 模型

### 2.3 频率限制（Rate Limiting）

- [ ] 服务端实现请求频率限制：
  - 全局：每 IP 每分钟 10 次 AI 调用
  - 每会话：最多 20 次 AI 调用（环境分析 + 错误诊断 + 修复建议）
  - 每日上限：免费用户 50 次 AI 调用 / 天
- [ ] 超限返回友好提示，引导升级付费
- [ ] 实现滑动窗口算法，避免突发流量

### 2.4 防伪造与防注入

- [ ] 服务端校验 `error.occurred` 消息的合理性：
  - 必须在 `step.execute` 之后才能触发
  - 同一步骤最多允许 3 次错误报告
  - 错误内容长度限制（防止用 stderr 传入大量文本让 AI 分析）
- [ ] 限制 AI prompt 注入：
  - 对客户端传来的 stdout/stderr 做内容清洗
  - 限制单次传入的文本长度（如 4KB）
  - 过滤明显的 prompt injection 模式
- [ ] 协议层增加消息序号（sequence number），防止重放攻击

### 2.5 AI 调用安全

- [ ] 服务端在调用 AI 前增加前置检查：
  - 检查当前会话的 AI 调用次数是否超限
  - 检查消息流是否符合正常安装流程（状态机验证）
  - 对 AI 输出做后置过滤（防止泄露系统 prompt）
- [ ] AI 流式响应不再直传原始 token，改为服务端解析后发送结构化结果
- [ ] 记录每次 AI 调用的 token 消耗，按会话/用户汇总

## 三、成本控制优化

### 3.1 智能缓存

- [ ] 安装计划缓存：
  - 相同环境特征 + 相同软件 = 缓存安装计划
  - 缓存 key：`hash(os + arch + runtime_versions + package_managers + software)`
  - TTL：7 天
  - 命中缓存时不调用 AI，成本为零
- [ ] 错误诊断缓存：
  - 相同错误签名 = 缓存修复建议
  - 缓存 key：`hash(error_message + exit_code + step_id)`
  - TTL：3 天

### 3.2 分级 AI 调用

- [ ] 简单场景走规则引擎，不调用 AI：
  - 已知错误模式（`common-errors.ts` 中已有）直接匹配返回
  - 标准环境 + 标准安装 → 使用预设计划模板
- [ ] 按场景分级调用不同模型：
  - 环境分析：轻量模型（便宜 10x）
  - 安装计划生成：轻量或中等模型（视复杂度）
  - 错误诊断：中等模型（需要推理能力）
  - 使用指导（tutor）：轻量模型（对话性质，量大）
- [ ] 实现 AI 成本仪表盘，实时监控 token 消耗

### 3.3 多厂商大模型支持（重要）

当前仅支持 Anthropic Claude，需要扩展为多厂商统一接入，实现成本最优和容灾切换。

#### 支持的厂商与模型

| 厂商 | 轻量模型（便宜） | 中等模型（主力） | 强力模型（复杂任务） |
|------|------------------|------------------|---------------------|
| Anthropic | Claude Haiku | Claude Sonnet | Claude Opus |
| OpenAI | GPT-4o mini | GPT-4o | o1 / o3 |
| Google | Gemini Flash | Gemini Pro | Gemini Ultra |
| DeepSeek | DeepSeek-V3 | DeepSeek-R1 | — |
| 阿里 | Qwen-Turbo | Qwen-Plus | Qwen-Max |

#### 架构设计
- [ ] 抽象统一的 `AIProvider` 接口：
  ```typescript
  interface AIProvider {
    name: string;
    chat(messages, options): AsyncIterable<string>;  // 流式输出
    estimateCost(inputTokens, outputTokens): number;
  }
  ```
- [ ] 实现各厂商 Provider：
  - [ ] `AnthropicProvider`（现有 `agent.ts` 重构）
  - [ ] `OpenAIProvider`（兼容 OpenAI SDK）
  - [ ] `GoogleProvider`（Gemini API）
  - [ ] `DeepSeekProvider`（兼容 OpenAI 接口格式）
  - [ ] `QwenProvider`（阿里云 DashScope，国内备选）
- [ ] 配置化模型路由（`ai-config.json` 或环境变量）：
  ```json
  {
    "routing": {
      "envAnalysis": { "provider": "deepseek", "model": "deepseek-v3" },
      "planGeneration": { "provider": "anthropic", "model": "claude-haiku" },
      "errorDiagnosis": { "provider": "anthropic", "model": "claude-sonnet" },
      "tutor": { "provider": "deepseek", "model": "deepseek-v3" }
    },
    "fallback": { "provider": "openai", "model": "gpt-4o-mini" }
  }
  ```

#### 智能路由策略
- [ ] 按场景复杂度自动选模型（简单→便宜，复杂→强力）
- [ ] Fallback 机制：主模型超时/报错时自动切换备用厂商
- [ ] 按成本排序：相同能力下优先选便宜的
- [ ] 服务器部署在日本，统一路由，不区分客户端地区

#### 环境变量扩展
```env
# 多厂商 API Key（按需配置，至少配一个）
ANTHROPIC_API_KEY=sk-ant-xxx
OPENAI_API_KEY=sk-xxx
GOOGLE_AI_API_KEY=xxx
DEEPSEEK_API_KEY=sk-xxx
DASHSCOPE_API_KEY=sk-xxx

# 默认 provider（可选，默认 anthropic）
AI_DEFAULT_PROVIDER=anthropic
```

### 3.4 Token 优化

- [ ] 压缩发送给 AI 的 context：
  - 环境信息只发送关键字段，不发送冗余数据
  - stdout/stderr 截断到关键片段（最后 50 行）
  - 对话历史限制最近 5 轮
- [ ] 使用更紧凑的 prompt 模板，减少每次调用的 input tokens

- [ ] 有新版本时提示用户，提供一键更新命令
- [ ] 可通过 `--skip-update-check` 跳过检查

### 4.6 协议版本兼容

客户端一旦分发就无法强制升级，服务端迭代协议后必须兼容旧客户端。

- [ ] WebSocket 握手时客户端上报协议版本号：`X-Protocol-Version: 1`
- [ ] 服务端按版本号走不同消息处理逻辑
- [ ] 不兼容的旧版本 → 返回强制更新提示并断开连接
- [ ] 兼容的旧版本 → 正常服务，新字段用默认值填充
- [ ] 版本号规则：小版本向后兼容（1.0 → 1.1），大版本可不兼容（1.x → 2.0）

### 4.7 客户端离线/降级模式

服务器不可用时客户端不应完全失灵。

- [ ] 环境检测结果始终可用（纯本地，不依赖服务端）
- [ ] 连接失败时展示离线模式提示，显示环境检测结果供用户自行参考
- [ ] 缓存上次成功的安装计划到本地，离线时可查看（不执行）
- [ ] 重连策略：指数退避重试，3 次失败后提示用户检查网络或稍后再试

## 六、错误处理与容错机制（优先级：P1）

### 6.1 服务端错误处理增强

#### 6.1.1 分层错误处理

- [ ] **WebSocket 层错误**：
  - 连接超时、断线重连失败 → 返回友好提示
  - 无效消息格式 → 返回格式错误详情（不直接断开连接）
  - 协议版本不兼容 → 返回升级提示 + 下载链接

- [ ] **AI 调用层错误**：
  - API Key 无效 → 记录日志 + 返回服务配置错误（不暴露 Key）
  - API 超时（30s+） → 自动 fallback 到备用 provider
  - API 限流（429） → 等待重试 or 切换 provider
  - AI 返回格式异常 → 使用预设模板兜底

- [ ] **业务逻辑层错误**：
  - 知识库加载失败 → 降级为通用安装方案
  - 计划生成失败 → 返回错误诊断 + 建议用户提供更多信息
  - 数据库连接失败 → 降级为无认证模式（仅基础功能）

#### 6.1.2 错误日志与追踪

- [ ] **结构化错误日志**：
  ```typescript
  logger.error({
    requestId,
    sessionId,
    errorCode: 'AI_TIMEOUT',
    provider: 'anthropic',
    model: 'claude-sonnet',
    duration: 32000,
    message: 'AI request timeout',
    stack: err.stack,
  });
  ```

- [ ] **错误分类与统计**：
  - 按错误类型（网络、AI、业务）分类计数
  - 高频错误自动告警（如 AI 失败率 > 10%）
  - 每日错误报告汇总发送给管理员

- [ ] **用户友好的错误消息**：
  - 技术错误 → 翻译为用户可理解的文案
  - 提供具体的修复建议（如"检查网络连接"、"稍后重试"）
  - 附带帮助链接（如 FAQ、GitHub Issues）

#### 6.1.3 AI 调用容错

- [ ] **重试机制**：
  - 临时性错误（超时、网络抖动）→ 指数退避重试 3 次
  - 永久性错误（API Key 无效）→ 不重试，直接 fallback

- [ ] **Fallback 链**：
  ```
  主 Provider (Claude Sonnet)
    ↓ (失败)
  备用 Provider 1 (DeepSeek)
    ↓ (失败)
  备用 Provider 2 (GPT-4o mini)
    ↓ (全部失败)
  预设模板（零 AI，规则引擎）
  ```

- [ ] **部分成功处理**：
  - 环境分析失败 → 使用默认分析，继续生成计划
  - 计划生成部分失败 → 返回已生成的步骤 + 警告

### 6.2 客户端错误处理增强

#### 6.2.1 网络错误处理

- [ ] **连接失败**：
  - 显示详细错误原因（无法解析域名、连接超时、证书错误）
  - 自动检测网络可用性（ping 公共 DNS）
  - 提供离线模式选项（仅环境检测）

- [ ] **消息发送失败**：
  - 消息队列机制：连接恢复后自动重发
  - 失败消息持久化到本地临时文件
  - 重连成功后从队列重新发送

#### 6.2.2 执行错误处理

- [ ] **命令执行超时**：
  - 显示超时提示 + 当前输出
  - 询问用户是否继续等待 or 终止
  - 记录超时命令，用于后续优化超时阈值

- [ ] **权限错误**：
  - 检测到 `permission denied` → 提示需要 sudo
  - 自动建议加 sudo 重试（需用户确认）
  - macOS 提示使用管理员账户运行

- [ ] **依赖缺失错误**：
  - 检测 `command not found` → 提示安装缺失依赖
  - 自动生成依赖安装命令（如 `brew install xxx`）

#### 6.2.3 用户体验优化

- [ ] **错误消息国际化**：
  - 支持中英文错误提示
  - 根据系统语言自动切换
  - 技术细节折叠，默认显示简洁版

- [ ] **错误上报**（可选，需用户同意）：
  - 崩溃时询问是否上报错误日志
  - 匿名化敏感信息（路径、用户名）
  - 上报到 Sentry 或自建错误收集系统

### 6.3 安装流程容错

#### 6.3.1 步骤失败恢复

- [ ] **自动重试策略**：
  - 网络相关步骤（下载）→ 失败后自动重试 3 次
  - 幂等步骤（检测版本）→ 失败后立即重试
  - 非幂等步骤（写文件）→ 不自动重试，询问用户

- [ ] **跳过与继续**：
  - 非关键步骤失败 → 询问是否跳过继续
  - 关键步骤失败 → 必须修复或中止

- [ ] **回滚机制**：
  - 步骤执行前创建快照（记录文件变更）
  - 失败后提供一键回滚选项
  - 回滚脚本自动生成（如删除新建文件、恢复备份）

#### 6.3.2 AI 修复建议交互

- [ ] **多方案选择**：
  - 展示 AI 建议的 1-3 个修复方案
  - 每个方案显示：描述、命令预览、置信度、风险等级
  - 用户选择后自动执行 or 手动执行

- [ ] **方案执行反馈**：
  - 执行修复方案后自动重试原步骤
  - 成功 → 继续下一步
  - 仍失败 → 再次请求 AI 诊断（限最多 3 轮）

## 八、v1.1 缓存与 v2.0 自助平台的演进路径

### 8.1 智能缓存系统（v1.1）

当前 v1.1 的缓存方案（见 3.1）采用**服务端集中式缓存**：

- 安装计划缓存 key：`hash(os + arch + runtime + package_managers + software)`
- 错误诊断缓存 key：`hash(error_message + exit_code + step_id)`
- 缓存存储：内存 LRU Cache + Redis（可选）
- 缓存更新：管理员手动刷新 or TTL 过期

**局限性**：
- 所有软件共享一个缓存池，软件提供方无法自主管理
- 知识库更新后缓存无法自动失效
- 无法按软件/版本维度查看缓存命中率

### 8.2 v2.0 自助平台的缓存升级

v2.0 实现软件提供方自助管理后，缓存系统需要**按软件隔离 + 自动失效**：

#### 8.2.1 分层缓存架构

```
全局缓存层（通用规则）
  ├── 常见错误规则（command not found, permission denied 等）
  └── 平台通用步骤（brew update, apt update）

软件级缓存层（软件提供方可控）
  ├── openclaw 的安装计划缓存
  ├── nodejs 的安装计划缓存
  └── 知识库查询缓存（按软件隔离）

版本级缓存层（未来扩展）
  ├── openclaw@1.0 的安装计划
  └── openclaw@2.0 的安装计划（不同安装方式）
```

#### 8.2.2 缓存自动失效机制

- [ ] **知识库更新触发**：
  - 软件提供方在后台编辑知识库 → 触发 webhook
  - 自动清除该软件的所有缓存（安装计划 + 错误诊断）
  - 下次安装时重新调用 AI 生成

- [ ] **版本化缓存**：
  - 缓存 key 中加入知识库版本号
  - 知识库更新后版本号递增，自然失效旧缓存

- [ ] **分软件的缓存统计**：
  - 软件提供方可查看自己软件的缓存命中率
  - 帮助评估知识库质量（命中率高 = 知识库完善）

#### 8.2.3 向后兼容

v1.1 的缓存实现需要**预留扩展位**，方便 v2.0 升级：

- [ ] 缓存 key 设计预留 `softwareId` 字段（v1.1 暂时固定值）
- [ ] 缓存存储接口抽象（`CacheProvider`），v2.0 可替换为 Redis Cluster
- [ ] 缓存统计数据表预留 `softwareId` 外键

**实现建议**：
```typescript
// v1.1 缓存 key（暂时 softwareId 固定）
const cacheKey = `plan:${softwareId}:${envHash}`;  // softwareId="openclaw"

// v2.0 升级后（softwareId 动态）
const cacheKey = `plan:${softwareId}:${version}:${envHash}`;  // 支持多版本
```

#### 涉及的新增组件
- **项目方门户（Web）**：项目方注册/登录、软件管理、知识库编辑、数据看板
- **知识库引擎升级**：支持在线索引更新、多软件隔离、版本化向量库
- **审核流程**：管理员审核提交的安装配置，防止恶意脚本
- **安装配置 DSL**：定义一套声明式安装描述格式（替代当前代码级的 planner）

#### 新增数据表（预留）
- `software_publishers` — 项目方/发布者账号
- `software_versions` — 软件版本与安装配置
- `knowledge_documents` — 知识库文档（关联软件和版本）
- `software_reviews` — 审核记录
- `install_statistics` — 按软件/版本/平台的安装统计

> **为什么放 v2.0**：这是平台化方向的核心功能，工作量大，且 v1.1 还没有足够的用户基础。
> 先用 v1.1 的 `SoftwareConfigModel` + 手动维护知识库跑通流程，积累经验后再做自助平台。
