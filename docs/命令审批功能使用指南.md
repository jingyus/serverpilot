## 📋 功能概述

命令审批机制解决了"危险命令无法使用"的问题，允许用户在需要时批准 RED/CRITICAL 风险级别的命令，同时保持安全性。

### 核心流程

```
用户发起危险命令（如 rm -rf /tmp/*）
  ↓
Server 识别风险级别（RED）
  ↓
创建审批请求 & SSE 推送通知
  ↓
Dashboard 弹窗显示命令详情
  ↓
用户点击"批准"或"拒绝"
  ↓
Server 收到决策 → 继续执行 或 拒绝
```

## 🗄️ 数据库 Schema

### `command_approvals` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | TEXT | 主键 UUID |
| `user_id` | TEXT | 发起用户 |
| `server_id` | TEXT | 目标服务器 |
| `command` | TEXT | 待审批命令 |
| `risk_level` | TEXT | 风险级别（red/critical/forbidden） |
| `status` | TEXT | 状态（pending/approved/rejected/expired） |
| `reason` | TEXT | 风险原因说明 |
| `warnings` | JSON | 警告信息数组 |
| `requested_at` | INTEGER | 请求时间 |
| `expires_at` | INTEGER | 过期时间（5分钟） |
| `decided_at` | INTEGER | 决策时间 |
| `decided_by` | TEXT | 决策人 |
| `execution_context` | JSON | 执行上下文（taskId/sessionId等） |

**Migration**: `packages/server/src/db/migrations/0014_command_approvals.sql`

## 🔌 API 端点

### 1. **创建审批请求** (由 TaskExecutor 自动调用)

```http
POST /api/v1/approvals
Authorization: Bearer <token>
Content-Type: application/json

{
  "serverId": "server-123",
  "command": "rm -rf /tmp/old_*",
  "riskLevel": "red",
  "reason": "Deleting files in /tmp directory",
  "warnings": ["Directory deletion", "Recursive operation"],
  "executionContext": {
    "sessionId": "session-456"
  }
}
```

**响应**:
```json
{
  "approval": {
    "id": "approval-789",
    "userId": "user-001",
    "serverId": "server-123",
    "command": "rm -rf /tmp/old_*",
    "riskLevel": "red",
    "status": "pending",
    "expiresAt": "2026-02-20T10:05:00.000Z",
    ...
  }
}
```

---

### 2. **获取待审批列表**

```http
GET /api/v1/approvals?status=pending
Authorization: Bearer <token>
```

**响应**:
```json
{
  "approvals": [
    {
      "id": "approval-789",
      "command": "rm -rf /tmp/old_*",
      "riskLevel": "red",
      "status": "pending",
      "requestedAt": "2026-02-20T10:00:00.000Z",
      "expiresAt": "2026-02-20T10:05:00.000Z"
    }
  ],
  "total": 1
}
```

---

### 3. **批准/拒绝命令**

```http
POST /api/v1/approvals/{approvalId}/decide
Authorization: Bearer <token>
Content-Type: application/json

{
  "decision": "approve"  // 或 "reject"
}
```

**响应**:
```json
{
  "approval": {
    "id": "approval-789",
    "status": "approved",
    "decidedAt": "2026-02-20T10:02:00.000Z",
    "decidedBy": "user-001"
  }
}
```

---

### 4. **SSE 实时推送**

```http
GET /api/v1/approvals/stream
Authorization: Bearer <token>
```

**事件流**:

```
event: connected
data: {"status":"connected"}

event: approval
data: {"id":"approval-789","command":"rm -rf /tmp/*",...}

event: decision
data: {"id":"approval-789","status":"approved"}

event: ping
data: {"timestamp":"2026-02-20T10:00:30.000Z"}
```

## 🚀 使用示例

### 场景 1: AI 对话中执行危险命令

1. **用户在 Chat 中输入**:
   ```
   清理 /tmp 下超过 7 天的文件
   ```

2. **AI 生成计划**:
   ```bash
   find /tmp -type f -mtime +7 -delete
   ```

3. **TaskExecutor 检测到 RED 风险级别**:
   - 自动调用 `CommandApprovalService.checkApproval()`
   - 创建审批请求
   - SSE 推送通知到 Dashboard

4. **Dashboard 弹窗显示**:
   ```
   ⚠️ 危险命令需要批准

   命令: find /tmp -type f -mtime +7 -delete
   风险级别: RED (高危)
   原因: 文件删除操作

   [批准] [拒绝]
   ```

5. **用户点击"批准"**:
   - 调用 `POST /api/v1/approvals/{id}/decide`
   - `CommandApprovalService` 收到决策
   - TaskExecutor 继续执行命令

6. **Chat 中显示执行结果**:
   ```
   ✅ 已删除 42 个文件
   ```

---

### 场景 2: 定时任务执行危险命令

1. **用户创建定时任务**:
   ```
   每天凌晨 2 点清理日志：rm -f /var/log/*.old
   ```

2. **凌晨 2 点触发**:
   - TaskExecutor 检测到 RED 风险
   - 创建审批请求（但用户不在线）
   - 5 分钟后自动过期
   - 任务标记为失败："Approval request expired"

3. **解决方案**:
   - **方式 1**: 用户在任务设置中添加命令到白名单
   - **方式 2**: 使用更安全的命令（如 logrotate）

## 🔐 安全特性

### 1. 自动过期机制

- 审批请求有效期：**5 分钟**
- 过期后状态自动变为 `expired`
- 避免长期等待和安全隐患

### 2. 用户身份验证

- 只有命令发起人可以批准自己的请求
- 防止权限提升攻击

### 3. 风险级别分级

| 级别 | 行为 | 示例 |
|------|------|------|
| **GREEN** | 直接执行 | `npm install`, `git clone` |
| **YELLOW** | 创建快照 + 执行 | `npm install -g` |
| **RED** | 需要审批 | `rm -r`, `dd if=/dev/zero` |
| **CRITICAL** | 需要审批 | `rm -rf /usr`, `mkfs` |
| **FORBIDDEN** | 绝对禁止 | `rm -rf /`, fork bomb |

### 4. 审计日志

- 所有审批请求记录在 `command_approvals` 表
- 包含完整的命令、决策时间、决策人
- 支持后续审计和分析
