# 文件上传与 FTP 功能设计方案

> 云版本与开源版本统一能力，贴合「智能化」宗旨：在平台内完成、少装额外工具、可选 AI 辅助。  
> **参考**：宝塔面板的「文件」模块——集成文件浏览、上传/下载、新建、搜索、收藏、终端、回收站等，一站替代 FTP 客户端。

---

## 一、参考：宝塔面板「文件」能力

参考宝塔等面板的「文件」模块，将以下能力纳入设计，实现**集成式文件管理 + 上传下载**，用户无需再单独使用 FTP 客户端：

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **上传/下载** | 工具栏「上传/下载」：本地上传至当前目录、下载选中文件到本地 | P0 |
| **面包屑 + 多标签** | 路径面包屑（如 `根目录 > home > www`），支持多标签页打开不同目录 | P0 |
| **前进/后退/根目录** | 导航栏：后退、根目录、前进，与面包屑联动 | P0 |
| **文件列表** | 表格式列表：勾选、名称（含图标/符号链接）、权限与所有者、大小、修改时间、备注、操作列 | P0 |
| **新建** | 「新建」下拉：新建文件、新建文件夹 | P0 |
| **搜索** | 按文件名/目录名搜索，可选「包含子目录」 | P1 |
| **文件内容搜索** | 在文件内容中搜索关键词（可选） | P1 |
| **收藏夹** | 常用路径收藏，快速跳转 | P1 |
| **终端** | 当前路径下打开终端（复用现有终端或 Web Shell） | P1 |
| **回收站** | 删除进回收站、恢复，避免误删 | P1 |
| **右键菜单** | 文件/目录右键：重命名、移动、复制、删除、权限、下载等 | P0 |
| **视图切换** | 列表视图 / 网格（图标）视图 | P1 |
| **磁盘占用** | 当前路径或根目录磁盘占用展示（如 40.9 GB） | P1 |
| **分页** | 大量文件时分页（如 500 条/页） | P1 |

上述与宝塔类似的布局与交互，便于用户从其他面板迁移到 ServerPilot，同时保留我们通过 Agent 执行、无需在服务器装 FTP 的差异化实现。

---

## 二、现状与诉求

- **现状**：无论云版还是开源版，都缺少「向服务器上传文件」的一体化能力；用户需要自行安装 FTP/SFTP 客户端（如 FileZilla）或在服务器上配置 vsftpd 等，割裂且不够智能。
- **诉求**：在 ServerPilot 内完成上传与基础文件管理，减少额外工具；若需传统 FTP/SFTP，也由平台「一键开通」而非用户手动安装配置。

---

## 三、设计原则（贴合智能化）

1. **平台内闭环**：优先在 Dashboard + Agent 通道完成上传，不强制用户在服务器上再装 FTP 服务。
2. **自动化优先**：若提供 FTP/SFTP 服务，则通过 Skill 或「服务管理」一键安装与配置，并生成连接信息。
3. **AI 可参与**：路径建议（如「网站根目录」「nginx 配置目录」）、自然语言触发上传、或生成 rsync/scp 命令供高级用户使用。
4. **统一体验**：云版与开源版使用同一套「文件管理 / 上传」入口，仅权限与配额按版本区分。

---

## 四、方案概览

| 能力 | 说明 | 适用场景 | 优先级 |
|------|------|----------|--------|
| **Web 文件上传** | Dashboard 选服务器 + 路径，上传文件；后端经 Agent 在目标机写文件 | 小/中文件、配置文件、脚本 | P0 |
| **Web 文件浏览** | 列出目录、查看文件内容（复用现有 read_file / list_files） | 确认路径、简单编辑前查看 | P0 |
| **一键开通 SFTP** | 在目标服务器上配置 SFTP（OpenSSH 已有则仅开放/说明），或安装 vsftpd；Dashboard 展示连接信息 | 大文件、批量上传、习惯 FTP 客户端 | P1 |
| **AI 辅助** | Chat 中「把这段配置写到服务器」「生成一条 rsync 命令」等 | 提升效率、减少记路径 | P1 |

## 五、详细设计

### 5.1 界面布局参考（宝塔风格）

- **左侧**：与现有 Dashboard 一致，保留「文件」为一级导航（与 网站、服务器、终端 等并列）。
- **顶部**：当前服务器标识 + 面包屑（如 `服务器A > /var/www`）+ 可关闭的标签页（多目录）。
- **工具栏**：`[ 后退 ] [ 根目录 ] [ 前进 ]`、`[ 上传/下载 ▼ ]`、`[ 新建 ▼ ]`、`[ 文件内容搜索 ]`、`[ 收藏夹 ▼ ]`、`[ 终端 ]`、当前路径磁盘占用、`[ 更多 ▼ ]`；右侧：搜索框（文件名/目录）+「包含子目录」、提示「文件操作可使用右键」、回收站、列表/网格视图切换。
- **主区域**：表格式文件列表（勾选、名称、权限/所有者、大小、修改时间、备注、操作）；底部：统计「共 N 个目录, M 个文件」+ 分页。

与宝塔对齐可降低学习成本；实现上我们仍通过 Agent 执行命令（list/read/write），不依赖服务器上的 FTP 服务。

### 5.2 Web 文件上传（P0）

**流程**：

1. 用户在 Dashboard 进入「文件」页，选择已连接的服务器与目标路径（可手动输入或通过「浏览」用现有 list_files 能力列出目录）。
2. 用户选择本地文件后点击上传；前端将文件分块或整块（小文件）发送到 Server。
3. Server 通过现有 **Agent 通道**（与 Skill 的 write_file 相同思路）在目标服务器上写入文件：
   - **文本/小文件**：可用现有 `executeCommand`，内容经安全转义后用 `printf` 或 `base64` 写入（与 `runner-executor` 的 `executeWriteFile` 一致）。
   - **较大文件 / 二进制**：用 `base64` 编码后通过 `echo <base64> | base64 -d > <path>` 写入，或分块多次写入再 `cat part1 part2 ... > path`，避免单次命令行过长。

**模块分工**：

| 模块 | 职责 |
|------|------|
| **Dashboard** | 新增「文件」页：服务器选择、路径输入/浏览、上传按钮、进度与结果提示；可选「AI 推荐路径」（调用 Chat 或单独接口）。 |
| **Server** | 新增 API：`GET /api/v1/servers/:id/files?path=`（list）、`GET /api/v1/servers/:id/files/content?path=`（read）、`POST /api/v1/servers/:id/files/upload`（multipart 或 JSON base64）；内部调用现有 TaskExecutor.executeCommand / 或复用 Skill 的 write_file 执行器逻辑。 |
| **Agent** | 无需改协议；继续通过现有 `step.execute` 执行写文件命令。若单次内容过大，可约定分块 step（例如 step 带 `chunk_index`），由 Server 拼装。 |

**安全与限制**：

- 路径校验：禁止写系统关键路径（如 `/etc/shadow`、根目录等），可配置白名单（如 `/home`、`/var/www`、用户可配置的「允许根路径」）。
- 单文件大小限制：如 10MB 内走 Web 上传，更大提示「请使用 SFTP 或 rsync」。
- 审计：所有写文件操作记入 audit log（serverId, userId, path, size）。

---

### 5.3 Web 文件浏览与列表（P0）

- **列表**：已有 AI 工具 `list_files`，封装为 REST：`GET /api/v1/servers/:id/files?path=/var/www`，返回结构化列表（名称、是否目录、权限、所有者、大小、修改时间）；支持分页参数（page, pageSize）。
- **查看内容**：已有 `read_file`，封装为 `GET /api/v1/servers/:id/files/content?path=...`，用于在线查看或编辑前预览。
- **下载**：`GET /api/v1/servers/:id/files/download?path=...` 或通过 content 接口由前端触发下载。

界面与 5.1 一致：工具栏「上传/下载」中「上传」打开本地上传对话框、「下载」对选中项调用下载接口；列表支持多选、右键菜单（重命名、删除、下载、权限等）。

---

### 5.4 新建 / 重命名 / 删除 / 权限（P0）

- **新建文件/文件夹**：工具栏「新建」→ 输入名称 → 调用 API 在当前路径下创建（Agent 执行 `touch` / `mkdir`）。
- **重命名、删除、修改权限**：通过右键菜单或操作列调用对应 API（Agent 执行 `mv`、`rm`、`chmod`/`chown`）；删除可选「移入回收站」（在服务器上移至约定回收目录，如 `.Recycle_bin`）或直接删除。
- API：`POST .../files/mkdir`、`POST .../files/mkfile`、`PATCH .../files/rename`、`DELETE .../files`、`PATCH .../files/permissions` 等，内部统一走 Agent 命令执行。

### 5.5 一键开通 SFTP（P1）

**思路**：多数服务器已有 SSH，即已具备 SFTP（sftp 子协议）；无需再装 FTP 服务。平台可：

- **方案 A**：在 Dashboard「文件」页或服务器设置中提供「SFTP 连接信息」说明（主机、端口 22、用户名建议、密钥/密码方式），并提示「您可直接使用 FileZilla / 系统 SFTP 客户端连接」。
- **方案 B**：若用户明确需要「FTP」（非 SFTP），则提供 Skill「安装 vsftpd」：在目标机执行安装与最小配置，开放指定目录、生成随机密码或绑定现有用户，并在 Dashboard 展示 FTP 地址与凭证（或「请使用 SSH 用户 + 密码」）。

**智能化体现**：用户无需自己查文档装 vsftpd、改防火墙；由平台完成安装与安全配置（如仅允许被动模式、限制目录）。

---

### 5.6 AI 辅助（P1）

- **Chat 内上传**：用户粘贴内容或上传小文件后说「把这段写到服务器的 /var/www/app/.env」→ 调用 write_file 或上述上传 API，由 AI 解析路径与内容。
- **生成命令**：用户说「生成一条把我本机 ~/dist 同步到服务器的 rsync 命令」→ AI 根据服务器信息生成 `rsync -avz -e ssh ~/dist user@server:/path`，用户本地执行。
- **路径建议**：在 Web 上传时提供「常用路径」或 AI 建议（如「检测到 nginx，网站根目录可能为 /var/www/html」）。

## 六、实施建议

1. **Phase 1（P0）— 对齐宝塔「文件」核心**  
   - **Server**：  
     - `GET /api/v1/servers/:id/files?path=&page=&pageSize=`（列表，含权限/大小/时间）；  
     - `GET /api/v1/servers/:id/files/content?path=`（查看）、`GET .../files/download?path=`（下载）；  
     - `POST .../files/upload`（上传）、`POST .../files/mkdir`、`POST .../files/mkfile`、`PATCH .../files/rename`、`DELETE .../files`、`PATCH .../files/permissions`；  
     - 内部统一经 Agent 执行命令（复用/抽离 write_file、list_files、read_file 等逻辑）。  
   - **Dashboard**：新增「文件」页（参考 5.1 布局）：选服务器 → 面包屑 + 标签页 + 工具栏（上传/下载、新建、后退/根目录/前进）→ 表格式列表（多选、右键菜单：重命名、删除、下载、权限）→ 底部统计与分页。  
   - **安全**：路径白名单、单文件大小限制、危险路径禁止、审计日志。

2. **Phase 2（P1）**  
   - 搜索（文件名 + 可选内容搜索）、收藏夹、回收站、终端入口、磁盘占用、列表/网格视图、分页优化。  
   - 一键 SFTP：在「文件」页或服务器详情中增加「SFTP 连接说明」或 Skill「配置 FTP(vsftpd)」。  
   - AI：Chat 内 write_file、生成 rsync/scp 命令、路径建议。

3. **云版与开源版**  
   - 同一套 API 与前端；云版可在「文件」操作上做配额或权限（如仅 owner 可上传到生产路径），与现有 tenant/role 一致。

---

## 七、与现有架构的契合

- **Agent**：已支持 `step.execute` 执行任意命令；写文件即「执行一条写文件的命令」，无需新协议。
- **Skill**：已有 `write_file` 工具与 `executeWriteFile`（printf 写内容）；可抽成公共「在服务器上写文件」服务供 API 与 Chat 共用。
- **安全**：沿用现有命令校验与审计；写文件作为「高风险操作」可要求二次确认或仅限部分角色。

按上述设计，既能满足「不单独装 FTP 也能上传」的诉求，又保留一键开通传统 FTP/SFTP 的选项，并让 AI 参与路径与命令生成，贴合智能化宗旨。
